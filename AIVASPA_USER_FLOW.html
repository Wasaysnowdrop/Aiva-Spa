The complete user flow of AivaSpa — from visitor to lead.

A 24/7 AI receptionist for med spas. A single <script> snippet is embedded on a spa's website; the AI answers from the spa's approved knowledge base, captures leads, and notifies the owner by email/SMS. This page documents every page, route, API, and step a user takes through the system.

Next.js 16.2.7 · App Router Supabase · Postgres + Auth + Realtime OpenAI-compatible LLM Resend · Twilio · Google Calendar
01 · Overview

What is AivaSpa?

AivaSpa gives every med-spa a 24/7 AI receptionist. Three audiences use the system: med-spa owners (set up the AI, manage leads), website visitors (chat with the AI, get answers, book), and platform admins (system health, all spas/users). Below is a single-glance map of the entire system.

Who it's for
Med spas & aesthetic clinics
Core promise
24/7 lead capture (no humans needed)
Install time
~5 min (paste 1 script tag)
Pricing tiers
Trial · Starter · Pro
Primary surface
Spa's own website (iframe widget)
Hosting
Vercel + Supabase
One-liner: A 24/7 AI receptionist for med spas. A single <script> snippet embeds an AI chat widget on a spa's website; the AI answers from the spa's approved knowledge base, captures leads, and notifies the owner by email/SMS.
02 · User journey

The 11-step journey — from landing to first lead

The first 7 steps are taken by the spa owner setting up their AI. Steps 8–11 are taken by the end visitor on the spa's website, then looped back to the owner via notifications.

1
Owner · Public

Lands on the marketing site

Hits / — the animated landing page with features, pricing CTA, and testimonials.

Why this works: hero copy, animated widget preview, social proof (logos, testimonials), and a "Get started free" CTA. No signup wall on the value-prop.

src/app/page.tsx · src/components/landing/*

2
Owner · Auth

Signs up (email + password or OAuth)

Goes to /signup → Supabase Auth creates the user → a row is created in spa_settings (workspace) and a 14-day trial subscription.

  • After signup → redirected to /onboarding automatically
  • Auth boundary is enforced by proxy.ts (Next 16 middleware)

src/app/signup · src/app/actions/auth.ts · proxy.ts

3
Owner · Onboarding

Goes through 9-section AI setup interview

An LLM-driven assistant asks the owner questions in natural language, then structures the answers into the spa's knowledge base. Each turn writes a partial onboarding_kb_draft to auth.users.user_metadata so it can be resumed.

The 9 sections: business → services → pricing → faqs → guardrails → hours → branding → channels → review. (See detail below.)

/onboarding · src/lib/ai/setup-assistant.ts · src/lib/ai/setup-assistant-schema.ts

4
Owner · Onboarding end

AI writes the final KB + widget config

On "Finish", the merged KB is written to:

  • widget_config — single-row appearance, position, brand color, proactive settings
  • knowledge_services — list of services with prices
  • knowledge_faqs — approved Q&A pairs
  • knowledge_guardrails — what the AI must never say/do

Owner is redirected to /dashboard.

5
Owner · Dashboard

Opens the dashboard — empty state

First-time dashboard shows:

  • Big "Install your widget" CTA → jumps to /dashboard/guide
  • Empty KPIs (no leads yet)
  • Empty Leads inbox with onboarding hints
  • Empty Conversations view

src/app/dashboard/page.tsx · src/app/dashboard/guide/page.tsx

6
Owner · Widget install

Copies the embed snippet

From /dashboard/widget or the Guide, owner copies a 1-line script:

<script src="https://aivaspa.online/embed/<spaId>/loader" data-spa-id="<spaId>"></script>

Pastes it before </body> on their website. That's it.

A row in widget_installs tracks this install by widget_key.

src/app/embed/[spaId]/loader/route.ts · src/lib/widget/installs.ts

7
Owner · Settings (optional)

Wires up the rest of the toolbox

Most owners do these next:

  • Notifications — add their email/SMS recipients in /dashboard/settings → stored in notification_channels
  • Google Calendar — OAuth-connect in /dashboard/integrations for slot lookups & bookings
  • API keys — create aiva_live_… keys in /dashboard/settings to push leads from external systems
  • Webhooks — register HMAC-signed outbound URLs for lead.created etc.
  • Team — invite Owner/Manager/Staff/Receptionist
  • Knowledge base — keep refining the KB in /dashboard/knowledge-base
8
Visitor · Spa's website

A real visitor arrives at the spa's website

The loader script runs, fetches /api/widget/config?spaId=…, and injects a host <div> + a sandboxed <iframe> pointing to /embed/<spaId>?parent=….

checkEmbedAccess(spaId) gates access — checks the install is active and the subscription isn't locked. If denied → EmbedLock is rendered instead.

src/app/api/widget/config · src/lib/widget/access.ts · src/app/embed/[spaId]/page.tsx

9
Visitor · Chat

Visitor types a question

Every message is sent to POST /api/chat with { sessionId, message, history, lead?, consentGiven, sourceUrl, spaId }. The route:

  • Loads the spa's KB (cached, single-row read)
  • Builds a system prompt from the KB + guardrails
  • Calls the LLM (src/lib/ai/llm.ts — falls back to canned responses if OPENAI_API_KEY is empty)
  • Persists every turn to chat_sessions (Realtime-enabled)
  • Returns the reply JSON to the widget

src/app/api/chat/route.ts · src/lib/ai/conversation.ts

10
Visitor · Lead

Visitor gives name, phone, service, consent

As soon as the AI detects a lead is captured, createPublicLead() is called:

  • Phone/email are normalized and matched against existing leads (dedup)
  • If a duplicate exists → transcript is merged into the canonical lead; old row gets merged_into_id
  • Lead status defaults to new; consent_given is recorded

src/lib/leads/server.ts · src/lib/leads/dedup.ts

11
Owner · Notifications

Owner is pinged instantly (email + SMS)

dispatchLeadNotifications() reads the spa's notification_channels and fires email (Resend) + SMS (Twilio). Each attempt is logged to notification_logs with status, retried up to 3× with exponential backoff.

If the spa connected Google Calendar, POST /api/google-calendar/book creates the event.

In parallel, webhooks.fireEvent('lead.created') POSTs a signed payload to every customer URL (HMAC-SHA256, X-AivaSpa-Signature header).

The owner opens the dashboard, sees the lead in /dashboard/leads, and the live transcript in /dashboard/conversations.

src/lib/notifications/dispatch.ts · src/lib/webhooks/index.ts

04 · Onboarding in detail

The 9 sections the AI interview walks through

The onboarding is an LLM itself: runSetupAssistantTurn in src/lib/ai/setup-assistant.ts. Each section has a strict JSON-draft schema in setup-assistant-schema.ts. Partial progress lives in auth.users.user_metadata.onboarding_kb_draft, so users can resume.

1 · Business profile

The basics: spa name, website, location, vibe, target customer. Writes the canonical spa_settings row.

Example prompts: "What's the name of your spa?", "What's your website?", "Describe your ideal customer in one sentence."

2 · Services

Each service has a name, category, duration, and short description. Written to knowledge_services.

3 · Pricing

Optional price ranges, deposits, package deals. Stays inside the KB — never fabricated by the AI.

4 · FAQs

Approved Q&A pairs. The AI can only answer from these — see the retrieval step in src/lib/ai/retrieval.ts.

5 · Guardrails

Hard rules the AI must follow. E.g. "never claim a treatment cures X", "never quote exact medical advice", "always offer to book a consultation". Written to knowledge_guardrails.

6 · Working hours

Day-by-day open/close. Used for the "after hours" branch in src/lib/ai/working-hours.ts — the AI will collect the lead and tell the visitor someone will reply in the morning.

7 · Branding

Primary color, accent color, logo, position (bottom-right / bottom-left), proactive greeting. Written to widget_config.

8 · Notification channels

Email recipients and SMS phone numbers that will be pinged for new leads. Stored in notification_channels with a per-recipient list.

9 · Review & finish

The full KnowledgeBase draft is shown. Owner can edit, then "Finish" writes it to all knowledge_* tables and widget_config, and redirects to /dashboard.

05 · Widget lifecycle

From script tag to iframe chat

Everything that happens between "owner pastes the snippet" and "visitor sees a chat bubble".

1. The loader JS executes

src/app/embed/[spaId]/loader/route.ts

Tiny script (under 50KB) reads data-spa-id from the script tag (or infers from its own URL), then calls fetch('/api/widget/config?spaId=…').

2. Config is checked

src/app/api/widget/config/route.ts

Returns brand, color, position, proactive settings, FAQ count, service count. Public — no PII. CORS: *. In-memory cache: 60 s.

3. Access is gated

src/lib/widget/access.ts · checkEmbedAccess(spaId)

Looks up widget_installs.active and the owner's subscriptions row. Returns not_found / inactive_install / expired or ok. If locked: true in the config, the widget is cancelled entirely — no chat.

4. Iframe is injected

src/app/embed/[spaId]/page.tsx

Host <div> + sandboxed <iframe src="/embed/<spaId>?parent=…">. The iframe renders ChatFrame with the spa's brand. postMessage handles "open / close / toggle / refresh / destroy" via window.AivaSpa.

5. Visitor sends a message

POST /api/chat

Body: { sessionId, message, history[], lead?, consentGiven, sourceUrl, spaId }. Validated with chatRequestSchema.

6. The turn is processed

src/lib/ai/conversation.ts

Loads KB → builds prompt → calls LLM → returns reply. Also writes the turn to chat_sessions (Realtime-enabled) so the dashboard sees it live.

7. Lead is captured (if applicable)

src/lib/leads/server.ts

When the AI sees name + phone + service + consent in the conversation, createPublicLead() runs — dedup, merge, save, dispatch.

07 · Architecture

What runs where

The whole system fits into a single Next.js 16 app (with RSC + Route Handlers), a Supabase project, and a handful of third-party APIs.

N

Next.js 16 (Vercel)

App Router · React 19 · Tailwind v4. The marketing site, dashboard, embed, all API routes, and proxy.ts middleware live here.

S

Supabase

Postgres (the single source of truth) + Auth (SSR via @supabase/ssr) + Realtime (chat_sessions stream) + RLS on every table.

L

OpenAI-compatible LLM

Default base URL https://api.freemodel.dev/v1, model gpt-5.4-mini. Falls back to a canned-response engine if OPENAI_API_KEY is empty.

E

Resend (email)

New-lead notifications. Console-logs the payload if RESEND_API_KEY is empty — dev mode.

T

Twilio (SMS)

Same dispatch path. Console-logs if no key. Retries 3× with exponential backoff on failure.

G

Google Calendar

OAuth2 — used for slot lookups, booking, and storing the event back in google_calendar_events.

Subdomain routing (host-based)

proxy.ts reads the Host header and decides what the visitor sees:

HostBehavior
aivaspa.online (and www.)Landing, signup, login, customer /dashboard
admin.aivaspa.onlineTransparent rewrite to /admin/* (system health, all spas/users, audit). app_metadata.is_admin required.
<spa>.aivaspa.onlineIf subdomain is in custom_domains → serve that spa's widget. Otherwise fall through to landing.
chat.<customer-domain> (Pro plan)White-label widget on the customer's own domain (same custom_domains resolution).
08 · API surface

Public, widget-facing & server-to-server

Three authentication tiers, one consistent JSON contract.

Public (no auth) — used by the widget on the spa's site

EndpointMethodPurpose
/embed/[spaId]GETServer-renders the chat widget inside an iframe
/embed/[spaId]/loaderGETThe script the spa pastes on their site
/api/widget/configGETBrand, color, position, proactive settings (CORS *)
/api/chatPOSTVisitor message → AI turn + optional lead save
/api/leadsPOSTDirect lead save (used when widget pre-collects fields)

API-key authed (server-to-server)

EndpointMethodAuthScopePurpose
/api/v1/leadsGETBearer aiva_live_… or x-api-key—Discovery — lists supported events
/api/v1/leadsPOSTsameleads:writeCreate a lead from any external system. Validates name, phone, service, preferredTime, optional email. Fires lead.created webhook scoped to that user.

Google Calendar (OAuth-gated)

EndpointMethodPurpose
/api/google-calendar/authGETStart OAuth flow
/api/google-calendar/callbackGETExchange code, store google_calendar_settings
/api/google-calendar/statusGETIs the spa connected? Which calendar?
/api/google-calendar/disconnectPOSTRevoke token, clear settings
/api/google-calendar/settingsPOSTUpdate booking duration, hours, buffer, working days
/api/google-calendar/slotsGETFree slots for a given date
/api/google-calendar/bookPOSTCreate a calendar event for a lead; recorded in google_calendar_events
API key format: aiva_live_… (prod) or aiva_test_…. Plaintext is shown once at creation. The server stores SHA-256 hash + a key_prefix for display. Scopes: leads:write, leads:read, webhooks:read, webhooks:write.
